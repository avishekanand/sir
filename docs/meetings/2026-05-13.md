- Pull request management includes peer review and learning from shared code contributions (05:05).
  - Avishek suggested early contributors review each others PRs to understand design decisions.
  - The approach aims to build quality and familiarity through collaborative code review.

  - The weekly schedule supports onboarding, setting standards, and cross-team alignment.
  - Members with scheduling conflicts agreed to provide asynchronous updates via Slack.

### Benchmark Integration and Management

- Current focus is on integrating new benchmarks and setting up pipelines for end-to-end RAG system evaluation (48:34).
  - Rahul and Shuvam will focus on integrating various benchmarks into the repository.
  - Mandeep will lead running experiments and benchmarking using GPU resources.

- Discussion on branch strategy for benchmarks to avoid breaking main development (51:28).
  - Proposal to create a separate benchmark branch to isolate benchmark code from main.
  - Benchmarks would be merged into main only after successful testing.
  - Sub-branches under the benchmark branch may allow contributors to add benchmarks without interfering with mainline stability.
  - Minimal tests in lightweight environments like Colab would validate benchmarks before merging to the benchmark branch.

- Benchmark folder structure and configuration management ideas were shared for scalability (01:01:08).
  - Suggested organizing benchmarks into separate folders with YAML config files.
  - This modular approach facilitates easier testing and integration.

- The team will maintain a shared benchmark list with triage status and ownership to track progress and prevent duplication (05:32).
  - A BENCHMARK.md document will be created to catalog all candidate benchmarks.
  - Columns will track triage, assignment, and integration status.

### Internal Code Quality and PR Hygiene

- Wiki documentation outlines PR and code review guidelines focused on quality and clarity (12:19).
  - Guidelines include explaining the problem solved, changes made, testing, and reviewer focus.
  - The rules are flexible for new benchmarks or early-stage code but encourage thoughtful commits.
  - Aim to prevent quality degradation as the project scales.

- Testing strategy emphasizes selective test types depending on the code change (15:32).
  - Smoke tests may be skipped for benchmark additions.
  - Unit tests remain important to ensure code correctness.

- Continuous integration feedback loops were discussed to improve PR workflows (16:39).
  - Shuvam noted that Copilot suggestions are not re-triggered automatically after PR updates.
  - Avishek will investigate and fix this to ensure consistent CI validation.

- Collaborative wiki updates and PR-based changes to documentation were encouraged to keep standards current (27:54).
  - Removing or adding guidelines will be a collective decision to avoid overengineering.

### Cost Metrics and Benchmarking Strategy

- Need for a cost matrix capturing abstract cost dimensions such as dollar cost, latency, and energy was emphasized (29:35).
  - Avishek proposed mapping costs for components like LLM calls, BERT encoder calls, and retrieval calls.
  - These cost values would enable optimization of pipelines under budget constraints.
  - The goal is to create Pareto curves showing trade-offs between quality and cost.

- Shuvam suggested incorporating energy consumption metrics using power usage and token counts (30:24).
  - Measuring energy consumption with tools like pijoules can quantify wattage per operation.
  - Combining energy, token count, and speed into a convex cost function could reflect real resource usage.

- Challenges around justifying and standardizing cost ratios were discussed (32:51).
  - Cost ratios depend on hardware, geographic location, and optimization goals.
  - Dollar cost can be derived from cloud API prices but varies by CPU/GPU usage.

- The team agreed to survey existing literature and tools on cost measurement for IR and LLM systems (38:34).
  - Avishek assigned Mandeep to start a shared document collecting resources and hypotheses.
  - The Green IR groups energy and emissions measurement libraries were identified as useful references.

### Results Management and Reproducibility

- Current experiment grids aggregate results but lack standardization and scalability for detailed per-query outputs (18:12).
  - Per-query results are too large to commit directly; a shared database or Redis server was proposed.
  - Aggregated results are a minimum requirement; detailed schemas will evolve as the project matures.

- Reproducibility is challenged by LLM non-determinism and large result data sizes (22:08).
  - Exact replication of pipeline results is difficult due to model variability.
  - Metadata like model versions, snapshot sources, and configurations are critical for reproducibility.

- Shuvam proposed a shared database accessible by all contributors for storing and comparing results (23:39).
  - This central repository would support comparative analysis between different runs and parameters.
  - Setup and hosting details remain to be decided.
